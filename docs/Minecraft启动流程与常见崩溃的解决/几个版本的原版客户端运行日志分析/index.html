几个版本的原版客户端运行日志分析

为什么要看懂日志​

在解决Minecraft运行故障时，除崩溃报告外，运行日志是另一个分析Minecraft故障原因的有效途径。通过运行日志，我们可以得出故障发生的时机从而初步判断可能的故障原因。有时，日志中的报错能直接告诉我们故障的准确原因。

在哪里找到日志​

通常，日志文件被储存在游戏目录下的logs文件夹中，最近一次运行日志为latest.log，更早的日志可在名称类似于2022-12-17-7.log.gz的压缩包中找到。

如果游戏在本体启动前就发生了致命错误，那么此次运行很可能不会产生任何日志文件，而且有些严重的报错并不会出现在日志中文件中，但这两种情况下一些关键运行输出仍可在控制台输出和启动器中显示的日志中被找到。此时，我们应格外注意日志文件夹中的日志中显示的崩溃时机和修改时间与当前的运行是否匹配，因为不匹配的运行日志会对故障检修造成极大的误导。

怎样看懂日志​

随便选其日志中的一行，如下

[14:25:44] [Render thread/INFO]: Backend library: LWJGL version 3.2.2 build 10

可以将这样一行分为以下几个部分

[时间] [线程/日志类型]: 正文

其中，常见的线程有以下几种

  • Render thread，渲染线程，也是游戏的主线程。
  • Server thread，服务端线程，仅在运行单人游戏或开设局域网服务器等情形中存在。
  • Netty Local Client IO #编号，客户端网络线程。
  • Netty Server IO #编号，服务端网络线程。
  • Sound engine，声音线程。
  • Worker-Main-编号，用途较多，原版中多用于世界生成和客户端的一些杂项事务。

常见的日志类型有以下几种：

  • INFO：正常运行时输出的信息。
  • WARN：非致命性错误，如Mod因网络出现问题而无法检查更新，通常是无害的。
  • ERROR：严重错误，可能导致游戏无法正常运行，但不一定会引起崩溃。
  • FATAL：致命性错误，通常会导致游戏无法正常运行甚至崩溃。
  • DEBUG：调试信息，一般不会被输出。

一般而言，日志中很大的篇幅是可以忽略的，我们只需要关注少量的重点内容：

  1. 日志文件是否真正存在？
  2. 日志是不是空白的？
  3. Minecraft本体类是否能被正常加载？
  4. 本地库加载是否正常？
  5. 资源包是否能被正常加载？
  6. 内置服务端是否能正常启动？
  7. 玩家是否能正常地加入游戏？
  8. 玩家退出游戏的过程是否正常？
  9. 是否出现表明异常的日志类型？
  10. 如果发生了崩溃或卡死，那么它发生前游戏正在做什么？有无报错？（最为重要）

如果添加了Mod，还应尤其关注如下内容：

  1. Mod的类是否加载失败？
  2. 是否检出了Mod兼容性问题？
  3. Mixin注入是否正常？
  4. 如果崩溃或卡死，这发生前是否有Mod报出错误？
